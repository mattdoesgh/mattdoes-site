# CONTEXT

## Vault
The Obsidian source tree of `.md` notes (plus `attachments/`) that the site is built from. Nested vaults (subdirs with their own `.obsidian/`) are skipped.

## Note record
What the vault reader hands Intake: `{ rel, content, mtime }`. `rel` is the vault-relative path used in every error message; `mtime` exists only as the `date:` frontmatter fallback.
Names the fs/pure seam: everything after the Note record is deterministic.

## Intake
The pure module that turns Note records into the Content model. Owns all loud-failure validation (publish kinds, slug rules, date/tags/aliases shapes, duplicate routes, duplicate about), thought splitting, stable thought IDs, sorting, and the slug index.
Invariant: same records in, same model out — no clock, no fs, no env. Error mode: throws with messages that name the offending `rel`.

## Content model
Intake's output: articles + thoughts + about note + slug index, all carrying raw markdown (never HTML). Articles and thoughts are newest-first; thought IDs are assigned oldest-first from CT wall-clock timestamps.

## Emit
The `lib/emit/` directory that writes the Content model to a `distDir`: markdown rendering (Shiki, wikilinks, embeds against the slug index), page rendering, asset hashing, feeds, sitemap. One module per concern behind the single `emit()` entry (`index.js` orchestrates; `render.js` markdown→HTML, `assets.js` plumbing + minify/hash + Early Hints, `csp.js` inline-script hashes, `feeds.js` Atom/sitemap/robots — ADR 0009). Deterministic given its inputs — listening data is passed in by the entrypoint, never fetched here. Pages are rendered by the React design system: Emit imports `render*` functions from the built SSG bundle (`design-system/dist-ssg/ssg.js`) and writes their HTML strings — see Design system and ADR 0005. The `/listening/` rows are the one exception, still rendered server-side by `templates/rows.js` (the browser Row module) to keep them byte-equal for live updates (ADR 0001).

## Listening
The Last.fm-derived track data (recent scrobbles + playcount). The build-time snapshot (`lib/listening.js`) sources it from the deployed listening Worker's own `/api/listening/recent` route — the production Cloudflare Pages build has no Last.fm credentials, so the Worker (already serving live scrobbles from KV) is the only build-time source that reflects reality. Resolution order: fresh disk cache → Worker → Last.fm direct (only with creds) → stale cache → empty; `LISTENING_OFFLINE=1` skips the network steps for CI/test builds (ADR 0006). It is an input to Emit, not part of the Content model.
The total scrobble count is the recent-tracks `@attr.total` everywhere — the single source the live Worker also uses, so the static stat and the live-updated stat can't disagree. Both consumers decode through `lib/lastfm.js` — the pure wire-format codec (URL builders + decoders, no fetch/fs/config). It is bundled into the listening Worker, so the ADR 0002 redeploy rule applies to it. The Worker itself is a producer/reader split: the Listening poller keeps KV fresh, and the request path is a pure KV reader (ADR 0008), so the build-time `/api/listening/recent` source and its payload shape are unchanged.

## Listening poller
The single Last.fm writer inside the listening Worker — a Durable Object (`ListeningPoller`, singleton via `idFromName('singleton')`) that owns *when* the upstream is called. A self-rescheduling alarm (~25s) makes one `user.getrecenttracks` call, derives both the `/now` and `/recent` payloads from one decode, and writes both KV keys; on failure it writes nothing so the last good snapshot survives. Because it is the only caller, Last.fm volume is constant and independent of visitor traffic, and the request path (the pure KV reader) never blocks on or even knows about Last.fm. A cron watchdog (`[triggers]` in `wrangler.toml`) only arms the alarm if none is pending — liveness, never an upstream call. See ADR 0008.

## Thought
A micro-post split out of a daily note on `## HH:MM` headings (CT wall-clock). Carries a stable id `t-YYYYMMDD-HHMM` derived from its own timestamp.

## Row
The rendered form of one timeline entry — one renderer per content kind (article, thought, listening) plus a per-kind empty state, shared verbatim by every timeline surface: /blog/, the section listings, and the in-browser listening live updates. The homepage's compact feed is deliberately not a Row consumer; it is its own renderer.

## Edge transport
The shared Worker response machinery in `workers/lib/transport.js`: the JSON+CORS envelope, preflight, error cache policy (`errorJson`), fail-open KV access, and request helpers. Caching *policy* (TTLs, cache-control strings) stays per-Worker; only the envelope is shared. Editing it means redeploying all Workers (`npm run deploy:workers`) — see ADR 0002.

## Design system
The React + TypeScript component library in `design-system/` (`@mattdoes/ds`). One source of truth playing two roles: it renders **every public page** to static HTML at build time (`renderToStaticMarkup`) and it is the component set synced to Claude Design (claude.ai/design) for visual editing. The page components live in `design-system/ssg/pages/` (IndexPage, ArticlePage, BlogPage, ListingPage, AboutPage, ColophonPage), composed from shared chrome components in `src/components/` — `PageShell` (topbar/footer + the status-pill logic, including the manual `siteConfig.status` override), `IdentityRail` (the left "page meta" card), `ElsewhereLinks` (off-site links). The document shell — `<head>`, importmap, preloads, enhancement scripts (the old `templates/base.js`) — is `design-system/ssg/document.tsx` (`renderDocument`); and `design-system/ssg/index.tsx` exposes the `render*` functions Emit imports from the built bundle (`dist-ssg/ssg.js`, react external, resolved from `design-system/node_modules`). The head also carries two navigation/UX boot pieces: a synchronous `theme-boot.js` (applies the visitor's saved theme/accent before the stylesheet paints, so a navigation never flashes the default) and a static `<script type="speculationrules">` that prerenders/prefetches same-origin links on intent — allowed by the `'inline-speculation-rules'` CSP source; `static/nav-prefetch.js` is now just the `rel=prefetch` fallback for browsers without the API. Because a hash source in `script-src` disables that directive's keyword inline allowances (`'inline-speculation-rules'` included), the build hashes *every* inline `<head>` script together: `lib/emit/csp.js` (`injectInlineScriptCsp`) appends the `sha256` of the emitted importmap **and** the speculation rules to `script-src` in the dist `_headers`. Drop either hash and the strict CSP silently blocks that script — a dead importmap kills every module that imports through it, dead speculation rules kill prerender/prefetch (ADR 0001, ADR 0007). Because a prerendered document executes its scripts against a stale prefs snapshot, `static/tweaks.js` re-applies on `prerenderingchange`/`pageshow` — see ADR 0007. Components emit the same class names as the old templates and reuse `static/_shared.css` verbatim, so the look is unchanged; content still comes from the vault via Intake. The presentation layer is React; everything else (static output, Workers, CSP, enhancement scripts, feeds) is untouched — see ADR 0003 (decision) and ADR 0005 (the completed cutover). Fidelity is held to semantic+visual equivalence (not byte-identical) because React controls HTML serialization; during the migration `design-system/ssg/render.tsx` proved it by diffing React output against the original template for identical data (that parity harness has since been removed along with the templates). The superseded `templates/*.js` page modules have been removed after the production soak; `templates/{rows,_helpers,_assets}.js` stay (browser Row module + asset registry). There is deliberately **no `ListeningRow` design-system component**: the `/listening/` row is owned by `templates/rows.js` for the byte-equal live-update contract, so a React mirror would only drift from it unnoticed — `ArticleRow`/`ThoughtRow` already carry the shared `.row` look in the design surface (ADR 0005).

## design-sync
The pipeline that uploads `@mattdoes/ds` to a Claude Design project so the components are editable visually. Config + notes live in `design-system/.design-sync/`; the converter builds `ds-bundle/` (the `window.MattdoesDS` bundle + preview cards) from the package's compiled `dist/`. Re-syncs run from `design-system/`. Brand fonts (JetBrains Mono) are committed under `static/fonts/` and ship into the bundle as real woff2 files — see ADR 0004 for why (and the `stripFontFace` Vite plugin it needs).
